# mac/ — Voice Keyboard macOS app

SwiftUI + AppKit menu bar app that wraps the Go pipeline (`core/libvkb.dylib`)
via a thin C ABI. Built with xcodegen — `project.yml` is the source of truth,
`VoiceKeyboard.xcodeproj` is committed alongside it for fresh-clone-friendliness.

## Quick build

From this directory (`mac/`):

```bash
make build       # regenerates project if needed, then xcodebuild Debug
make run         # build + open the .app
make test        # SwiftPM tests for VoiceKeyboardCore
make project     # regenerate VoiceKeyboard.xcodeproj from project.yml
make clean       # nuke local DerivedData / .build
```

`make build` is incremental and cheap when nothing changed (~1.5s warm,
~2s after touching a Swift file). `make project` is a no-op unless
`project.yml`, `Configs/SharedSettings.xcconfig`, or any tracked
`.swift` source has changed (Make tracks the `.pbxproj` as a real
file target — see `Makefile`).

## Source of truth

- `project.yml` — xcodegen spec. Edit this for project-wide settings,
  build phases, source paths, dependencies. **Never** edit
  `VoiceKeyboard.xcodeproj/project.pbxproj` by hand; xcodegen will
  overwrite it on the next regen.
- `Configs/SharedSettings.xcconfig` — committed defaults that any
  contributor inherits (ad-hoc signing, no team needed).
- `Configs/DeveloperSettings.xcconfig` — gitignored optional override
  pulled in by SharedSettings via `#include?`. Use this for personal
  team-ID + Apple Development signing. Sample:
  `Configs/DeveloperSettings.xcconfig.sample`.
- `VoiceKeyboard.xcodeproj/` — tracked, regenerated by xcodegen.
  Determinism verified: 3 consecutive regens produce the same
  `.pbxproj` SHA so PR diffs from `make project` are clean.

## Editing build settings

| Where it lives | Edit via |
|---|---|
| Project-wide setting (deployment target, ARCHS, Swift version, build phases) | `project.yml`, then `make project` |
| Per-machine setting (your team ID, signing identity for personal builds) | `Configs/DeveloperSettings.xcconfig` (gitignored, optional) |
| Per-target setting | nested `settings.base` block in `project.yml` |

**Don't** change Build Settings via the Xcode IDE inspector — those edits
go straight into the `.pbxproj` and survive only until the next
`make project`. The post-merge git hook (see `scripts/git-hooks/`)
auto-runs `make project` after every `git pull`, so any IDE-only
tweaks get wiped silently. If you find yourself wanting to change a
build setting, edit `project.yml` and regenerate.

## Signing — open-source-ready defaults

`SharedSettings.xcconfig` ships:

```xcconfig
DEVELOPMENT_TEAM =
CODE_SIGN_STYLE = Manual
CODE_SIGN_IDENTITY = -
```

Translation: ad-hoc signing by default. Anyone — including someone
without an Apple Developer account — can clone, open the project,
and hit Run. Useful for the eventual public release.

If you have a team and want personal Xcode builds to use Apple
Development signing (so TCC permissions like the mic stay granted
across rebuilds), drop in your override once:

```bash
cp Configs/DeveloperSettings.xcconfig.sample Configs/DeveloperSettings.xcconfig
# edit Configs/DeveloperSettings.xcconfig with your DEVELOPMENT_TEAM
```

The override is gitignored, so it never gets committed. CI builds
ignore it (the file isn't on the runner) and ship ad-hoc.

## Build phases (project.yml)

1. **Build Go core** (preBuild) — `make build-dylib` in `../core/`.
   Sources `~/.zprofile` + `~/.bash_profile` so Xcode IDE builds
   find `make` / `go` on the user's PATH.
2. **Sources** — Swift compile.
3. **Copy libvkb.dylib + libdf.dylib into Frameworks** (postCompile)
   — `cp -p` preserves mtime + signature; the source dylibs already
   have the correct `@rpath/*` install_name.
4. **Copy TSE models into Resources (Release only)** — skipped on
   Debug; ModelPaths falls back to `~/Library/Application Support/`.
5. **Bundle Homebrew dylibs (Release only)** — walks the transitive
   ONNX + Whisper closure, copies, install_name_tool rewrites, and
   re-codesigns. Sentinel-guarded so warm Release rebuilds skip the
   80-second walk.

Debug builds skip steps 4 + 5 entirely — the dev runs against
Homebrew at `/opt/homebrew` directly. Release builds (CI, distribution)
do the full bundle for self-contained `.app`s.

## Common gotchas

- **`make project` is a no-op unless inputs changed.** If you suspect
  a stale project, force a regen with `rm -rf VoiceKeyboard.xcodeproj
  && make project`.
- **Editing in Xcode IDE doesn't persist** for project-level Build
  Settings (see "Editing build settings" above).
- **The post-merge hook expects `Configs/SharedSettings.xcconfig` to
  exist.** It always does because the file is tracked, but if you ever
  delete it during a refactor, `make project` will fail until you put
  it back.
- **TSE models** (`core/build/models/*.onnx`) aren't in git. Release
  builds bundle them into the .app's Resources via the postCompile
  phase in `project.yml`. Debug builds expect them at
  `~/Library/Application Support/VoiceKeyboard/models/`. Build them
  once via `./enroll.sh` from the repo root (the script also records
  voice for enrollment — Ctrl+C that step if you only want the
  models). Voice enrollment itself is now in-app via Settings →
  Voice → Enroll, which calls `vkb_enroll_compute` on the C ABI.
- **libvkb.dylib not built / cgo errors during `make build`.** The
  Xcode preBuild phase rebuilds `core/build/libvkb.dylib` on every
  build, but it can fail with opaque linker errors if Homebrew cgo
  deps are missing (`whisper-cpp`, `ggml`, `onnxruntime`). Run
  `./scripts/setup-dev.sh` from the repo root — it installs the deps
  and bootstraps the dylib so the first Xcode build is clean. The
  post-merge / post-checkout git hooks also rebuild the dylib when
  `core/*.go` changes, so a `git pull` keeps it in sync.
- **Signing certs aren't bundled with the app.** A Developer ID +
  notarization workflow for actual distribution is a separate effort
  — see `docs/distribution.md` (TBD) and the Build .app workflow's
  caveat about ad-hoc artifacts.
